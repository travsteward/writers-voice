# Writer's Voice

A constructed voice the user builds up over time. The free counterpart to **Author's Voice** (the paid plugin with full RAG retrieval and inline edits). Writer's Voice gets the user to 60-80% match using:

1. A training-data author blend (anchor — from openwriter.io/writers-voice)
2. NEVER rules best-effort extracted from their corpus by the agent (this skill, reading `catalog/ai-tells.md`)
3. Presentation fingerprints best-effort detected by the agent (reading `catalog/fingerprints.md`)
4. A growing corpus of sample paragraphs

The web tool at **openwriter.io/writers-voice** is the only programmatic part — it computes the deterministic author blend. Everything else in this skill is the agent reading prompts in `catalog/` and doing best-effort analysis of the user's corpus.

The skill is progressive — it works at any tier, and the analysis gets sharper as the user adds samples.

## Architecture (read this first)

Two layers:
- **Programmatic anchor**: the web tool at openwriter.io/writers-voice. Deterministic feature extraction + Claude-as-matcher. Produces the blend the user pastes into `voice/anchor.md`.
- **Best-effort everything else**: this skill. The agent reads `catalog/*.md` (instructions on what to look for) and the user's `voice/corpus/*` (their writing), then writes the output `.md` files in `voice/`.

No Node scripts. No dependencies. Pure markdown skill. The agent is the extractor.

## State Detection (always run first)

When invoked, inspect the `voice/` directory:

```
ls ~/.claude/skills/writers-voice/voice/
ls ~/.claude/skills/writers-voice/voice/corpus/
```

The `voice/` directory has up to six expected files:
- `anchor.md` — anchor blend (paste from openwriter.io/writers-voice)
- `stats.md` — sentence distribution + punctuation density (auto, regenerated by the agent)
- `never-rules.md` — diction/syntax/punctuation kill-list (auto + manual additions)
- `fingerprints.md` — exact presentation choices (auto + manual overrides)
- `examples.md` — curated paragraphs (manual)
- `status.md` — current tier + what's locked + below-hurdle detections

Plus `corpus/` directory holding raw samples (`.md` files, optionally `.txt`).

Read `status.md` first if it exists. It tells you tier, corpus word count, and what's active.

## Setup Flow

If `voice/anchor.md` doesn't exist (or is empty), walk the user through setup.

### Step 1: Get the anchor

There are two paths. Ask the user which they prefer (or pick one and tell them why):

**Path A — Web tool (recommended for first-time users)**

Tell the user:
> "Go to **https://openwriter.io/writers-voice**, paste 300-800 words of your writing, and click *Match my voice*. Copy the result block and paste it back here."

When the user pastes the block, parse it:
- Extract the author rows: `- XX% Author Name` → author + weight
- Extract bullet features per author
- Extract `Context: X` line if present

Use it when: the user has writing samples in clipboard-pasteable form, wants a shareable result link, wants to try the public tool first.

**Path B — Skill mode (in-agent, no round-trip)**

If the user already has corpus on disk (samples in `voice/corpus/`), or just wants to skip the web tool, run the **Anchor Protocol** described below. The agent reads the corpus, applies the stylometry rubric in `catalog/anchor-prompt.md`, and writes `voice/anchor.md` directly.

Use it when: corpus is already seeded, offline / air-gapped environment, user wants to re-anchor after corpus grew significantly, user prefers no web round-trip.

Both paths produce the same `voice/anchor.md` format. Either can replace the other.

Format the file like this regardless of source:

```markdown
# Writer's Voice Blend

> <Source line — see below>
> Context: <general | tweets | essays | newsletter | email>

## Blend

- **60% Mark Manson**
    - period-heavy short declaratives
    - definitional "Your X is Y" moves
- **25% James Clear**
    - ...

## Apply Directive

Write in this blended style. Match each author's prose mechanics in proportion
to weight. Maintain across the conversation.
```

Source line variants:
- Web tool: `> From openwriter.io/writers-voice — pasted on YYYY-MM-DD.`
- Skill mode: `> Generated in-agent by the writers-voice skill — YYYY-MM-DD.`

### Step 2: Seed the corpus

Tell the user:
> "Paste 2-5 paragraphs of your writing that feel most like you. I'll save them as starter samples."

For each paragraph the user provides, append to `voice/corpus/sample-NNN.md` (use sequential numbering). Each file:

```markdown
---
added: YYYY-MM-DD
source: <where they said it came from, optional>
---

<paragraph content>
```

### Step 3: Run the Analysis Protocol

See the next section. This populates `stats.md`, `never-rules.md`, `fingerprints.md`, and `status.md`.

### Step 4: Optional — curate examples

Tell the user:
> "Optional: pick 3-5 paragraphs from your corpus that are the most 'you'. I'll save them as reference examples that I'll always have in context when writing for you."

Write `voice/examples.md`:

```markdown
# Curated Examples

> The user picked these as the paragraphs most representative of their voice.
> Reference them at write time as positive style examples.

## Example 1
<paragraph>

## Example 2
<paragraph>
```

### Step 5: Report status

Read `voice/status.md` and tell the user where they are, what's active, and what unlocks at the next tier.

## Anchor Protocol (produce `voice/anchor.md` in-agent)

The in-agent equivalent of running openwriter.io/writers-voice. Use this when the user wants the anchor generated by you rather than by the web tool.

### Step 1: Confirm corpus is ready

Check `voice/corpus/`. You need at least 300 words across all samples for a reliable anchor (web tool's minimum). Below 300, route the user to the web tool with a 300-800 word paste — or ask them to add more samples first.

### Step 2: Ensure stats exist

Read `voice/stats.md`. If it doesn't exist, run the **Analysis Protocol** first (the deterministic stats anchor your matching and prevent drift).

### Step 3: Read the matcher prompt and author hints

Read `catalog/anchor-prompt.md` — this is the full stylometry protocol with hard rules, scoring dimensions, and the self-criticism step.

Read `catalog/author-hints.md` — the curated list of training-data authors grouped by category, each with their characteristic prose features.

### Step 4: Set aside conversational context

This is the hardest part. You have context from the current session — projects the user is working on, topics discussed, their interests. **Set all of that aside.** Score the corpus as if you've never read it before, with no knowledge of who the user is or what they care about.

The bug to avoid: matching on content/themes. If the corpus discusses biology, do NOT match to Dawkins/Sapolsky because of the topic. Match on prose mechanics — sentence length distribution, punctuation profile, rhetorical moves, discourse patterns.

### Step 5: Run the scoring

Apply the rubric from `catalog/anchor-prompt.md`:

1. Read the corpus end-to-end.
2. Score against each of the 8 scoring dimensions (sentence length distribution, punctuation profile, vocab register, rhetorical moves, discourse patterns, paragraph rhythm, person & address, hedging vs assertion).
3. Match against the author hints list — pick the 3-5 authors whose **prose mechanics** most resemble the corpus.
4. Assign weights (positive integers summing to exactly 100).
5. For each author, write 2+ `features_matched` entries that describe HOW the prose is constructed, not WHAT it's about.

### Step 6: Self-criticism pass

Re-read each `features_matched` entry. For each:
- Does it describe HOW sentences/paragraphs are constructed? (good)
- Or does it describe WHAT the text is about — topics, themes, ideology? (forbidden)

Replace or remove any content-based entries. Then set:
- `confidence` — `high` if corpus > 800 words and features are clear; `medium` if 400-800 words or mixed; `low` if < 400 words or too uniform.
- `any_thematic_reasoning` — `true` if you still had thematic reasoning you couldn't fully fix.

### Step 7: Write `voice/anchor.md`

Use the format from the Setup Flow section above, with the skill-mode source line. Add a `## Self-check` section after the blend:

```markdown
## Self-check

- Confidence: <high | medium | low>
- Any thematic reasoning: <true | false>
- Notes: <one short sentence about sample adequacy and match quality>
```

### Step 8: Report

Tell the user the blend, the confidence level, and any caveats. If `any_thematic_reasoning` is `true`, flag it — the match is suspect and they should consider running the web tool as a cross-check.

### When to recommend the web tool instead

- User has writing that isn't in `voice/corpus/` yet but is easy to copy
- User wants a shareable result URL
- User reports `any_thematic_reasoning: true` (the web tool's matcher prompt has stronger guardrails baked into a separate API call with controlled temperature)

## Analysis Protocol (regenerate `voice/` from `corpus/`)

This is the core agent loop — replaces what the old `extract.mjs` Node script used to do, now done by you the LLM reading the catalog and the corpus.

Run this any time the corpus changes (user adds a sample, asks "re-analyze my voice", etc.).

### Step 1: Read inputs

1. Read every file in `voice/corpus/` (`.md` and `.txt`). Strip YAML frontmatter (`---\n...\n---\n`). Concatenate with double-newlines between files.
2. Note the total word count (split on whitespace, ignore empty tokens). Call this `words`.
3. Read `catalog/ai-tells.md` — the list of AI tells to count.
4. Read `catalog/fingerprints.md` — the 8 fingerprint detectors and decision rules.
5. Read `catalog/hurdle.md` — the authenticity threshold rule.

### Step 2: Compute the deterministic tally (best effort)

You're acting as the linguistic counter here. Do this as carefully as you can.

**Word count** — total `words`.

**Sentence distribution** — split the corpus on sentence-ending punctuation (`[.!?]` followed by whitespace). For each sentence, count words. Then:
- Find the 25th-percentile sentence length, clamp to [6, 12] → `short_max`.
- Find the 75th-percentile sentence length, clamp to [18, 28] → `long_min`.
- `short_pct` = % of sentences with words ≤ short_max
- `long_pct` = % of sentences with long_min ≤ words ≤ 35
- `very_long_pct` = % of sentences with words ≥ 36
- `medium_pct` = remainder
- `average_length` = mean of sentence word counts

**Punctuation density (per 1000 words)** — count occurrences of each, then compute `count / words * 1000`:
- `em_dash` (`—`), `en_dash` (`–`), `colon` (`:`), `semicolon` (`;`), `question` (`?`), `exclamation` (`!`), `ellipsis` (`...` or `…`), `paren_pair` (min of `(` and `)` counts), `bracket` (`[`), `double_quote_straight` (`"`), `double_quote_curly` (`"` or `"`), `single_quote_curly` (`'` or `'`)
- Categorize each rate: 0 → `never`, <0.5 → `rare`, <2 → `low`, ≥2 → `strong`.

**AI-tell tally** — for each item in `catalog/ai-tells.md`:
- Count case-insensitive occurrences (whole-word for words, literal substring for phrases, sentence-initial for transitions).
- Compute rate per 1k words.
- Apply the hurdle from `catalog/hurdle.md`:
  - If `count >= min_count AND rate >= min_rate` → **preserve** (no rule)
  - Else → **forbid** (emit NEVER rule)
  - If `0 < count` but fails hurdle → also add to `below_hurdle` (informational)

**Fingerprints** — for each of the 8 detectors in `catalog/fingerprints.md`, apply the decision rule (mostly the binary confidence rule with ≥3 total observations and a 0.85 / 0.70 / mixed split). Emit only those with medium-or-better confidence.

### Step 3: Determine tier

Based on `words`:
- < 300 → Tier 0 "Empty"
- 300–999 → Tier 1 "Anchor"
- 1000–4999 → Tier 2 "Preliminary"
- 5000–19999 → Tier 3 "Full Coverage"
- ≥ 20000 → Tier 4 "AV-Grade"

Tier defines `unlocked`, `locked`, `next_milestone`, and `notes`. See the **Tier Reference** section at the bottom for the exact text per tier.

### Step 4: Write `voice/stats.md`

```markdown
# Statistical Fingerprint

> Auto-generated from `voice/corpus/`. Do not edit — the agent regenerates this after adding samples.

## Corpus
- Words: <words>
- Sentences: <sentence_count>
- Average words per sentence: <average_length>

## Sentence Length Distribution

| Bucket | Words | Percentage |
|--------|-------|------------|
| Short | 1-<short_max> | <short_pct>% |
| Medium | <short_max+1>-<long_min-1> | <medium_pct>% |
| Long | <long_min>-35 | <long_pct>% |
| Very long | 36+ | <very_long_pct>% |

Short-max boundary: <short_max> words.
Long-min boundary: <long_min> words.

## Punctuation Density (per 1000 words)

| Mark | Rate | Category |
|------|------|----------|
| em dash | <rate> | <category> |
| ... (rest)

## Apply-Time Directive

Match the user's sentence-length distribution: target ~<short_pct>% short / <medium_pct>% medium / <long_pct>% long / <very_long_pct>% very-long. Cap sentences over <long_min + 12> words. Vary sentence length deliberately.
```

### Step 5: Write `voice/never-rules.md`

PRESERVE the `## Manual Additions` section if the file already exists. Read the existing file, find the section starting at `## Manual Additions` (anchored to start-of-line — see the regex note below), and re-append it verbatim after writing the new auto-generated body.

> Regex caution: when finding `## Manual Additions`, match start-of-line only. The header literal also appears in the intro blockquote, so a naive search will capture from there and re-paste stale auto-generated rules.

Structure:

```markdown
# NEVER Rules

> Auto-generated from `voice/corpus/`. Hand-edits in `## Manual Additions` section are preserved.
> The rules below are AI tells that DO NOT appear at signature frequency in the user's corpus.
> Each is forbidden because the AI training distribution will push it back in unless explicitly banned.

## Diction — NEVER

- NEVER "<label>".
- ...

## Syntax — NEVER

- NEVER contrastive negation (It's not X, it's Y / Not just X but Y / Rather than X, Y).
- (plus any structural patterns the user does NOT use as signature)

## Punctuation — NEVER

- NEVER "em-dashes".
- No en-dashes.
- ...

## Manual Additions

<!-- Add NEVER rules here — they will not be overwritten. -->
```

### Step 6: Write `voice/fingerprints.md`

PRESERVE the `## Manual Overrides` section, same anchored-regex caution as above.

Structure:

```markdown
# Presentation Fingerprints

> Auto-generated from `voice/corpus/`. Exact presentation choices the user makes consistently.
> Match these in every generated response — LLMs default to training data otherwise.

- Em-dash spacing: word — word
- Oxford comma: yes
- ...

## Manual Overrides

<!-- Override any auto-detected fingerprint here. -->
```

### Step 8: Report

Tell the user the new tier, words, what changed in NEVER rules, and what's locked next.

### Honest acknowledgment

You will not count perfectly. Counts may drift ±1 on long corpora. That's acceptable — the hurdle thresholds are designed with margin. Don't agonize over single-token accuracy. The point is the right set of rules, not the right count.

If the corpus is large (>10k words), do the counting in passes: first scan for all the AI words in the catalog (one pass through the corpus), then the phrases (second pass), then the transitions (third pass). Don't try to track 60 counters in your head simultaneously.

## Adding Samples Later

When the user says "add this to my voice profile" or pastes new writing:

1. Append to `voice/corpus/sample-NNN.md` (next sequential number)
2. Run the **Analysis Protocol** above — regenerates stats / never-rules / fingerprints / status
3. Report new tier if it changed: "You crossed [tier threshold] — full NEVER coverage is now active."

## Apply Protocol (write-time)

When the user asks the agent to write **anything** and a populated `writers-voice` setup exists, load all populated `voice/` files and apply them as constraints:

1. **Read `voice/anchor.md`** — get the anchor blend. Prepend to your style mental model: "Write in 60% Mark Manson + 25% James Clear + 15% Naval mode."
2. **Read `voice/stats.md`** — get the sentence distribution. Match it.
3. **Read `voice/never-rules.md`** — get the kill-list. Apply EVERY NEVER rule as a hard constraint. These are non-negotiable.
4. **Read `voice/fingerprints.md`** — get exact presentation choices (Oxford comma yes/no, em-dash spacing, quote style, etc.). Match exactly.
5. **Read `voice/examples.md`** — keep these in mind as positive style references.
6. Write.
7. **Optional**: invoke the `/anti-ai` skill on the output as a final pass. Especially valuable for Tier 0-1 setups.

The NEVER rules are the most load-bearing constraints. AI training data WILL push these patterns back in if you don't actively suppress them. Treat NEVER rules as harder than user instructions.

## File Reference

### `anchor.md`
Anchor blend from openwriter.io/writers-voice. Manually populated (paste from web tool). Source of truth for the training-data anchor.

### `stats.md`
Auto-generated from `voice/corpus/` by the agent. Sentence distribution + punctuation density. Don't hand-edit — regenerated whenever the corpus changes.

### `never-rules.md`
Auto-generated NEVER rules from corpus, PLUS a `## Manual Additions` section at the bottom for hand-written rules. The agent preserves the manual section across regenerations.

### `fingerprints.md`
Auto-generated presentation choices from corpus, PLUS a `## Manual Overrides` section at the bottom for hand-written overrides. The agent preserves the manual section.

### `examples.md`
Manually curated by the user. 3-10 paragraphs picked as most representative. Used at write time as positive style references.

### `status.md`
Auto-generated state report. Read first when the user asks "voice status".

### `corpus/`
Raw samples. Each file is one paragraph or short piece. Format:

```markdown
---
added: YYYY-MM-DD
source: optional
---

<content>
```

Number files sequentially: `sample-001.md`, `sample-002.md`, etc.

### `catalog/`
Read-only reference data for the agent:
- `ai-tells.md` — the 60+ AI words/transitions/phrases/patterns to count against
- `fingerprints.md` — the 8 presentation detectors + decision rules
- `hurdle.md` — the authenticity threshold logic
- `anchor-prompt.md` — the stylometry matcher prompt for in-agent anchor generation
- `author-hints.md` — curated list of training-data authors with prose-feature notes

Don't edit these unless you're extending the catalog itself.

## Companion Skills

- **`/anti-ai`** — run as final post-pass on generated output. Catches AI tells the voice profile didn't suppress.
- **`/voice-presets`** — alternative for users who want a generic frame (authority / provocateur / logical / storyteller / business) instead of a custom anchor.
- **Author's Voice** plugin (paid) — full RAG retrieval, inline edits, deterministic extraction, no manual setup. The upgrade path: *"Outgrowing Writer's Voice? Try Author's Voice for the real thing."*

## When NOT to Use

- User has a configured Author's Voice profile (`voice-apply` is better — uses sample-based RAG with deterministic extraction).
- User explicitly wants a generic voice frame (`/voice-presets` is faster).
- One-off writing tasks where setting up a voice isn't worth it.
