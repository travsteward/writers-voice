# Writer's Voice

Public source for the **writers-voice** skill — a free, pure-markdown, agent-agnostic skill for constructing a voice for AI writing. Pairs with **Author's Voice** (paid OpenWriter plugin) as the free/paid split: same negative-first voice-profiling architecture, different distribution and capability tier.

## Dual Location Model

The skill exists in two places:

| Location | Purpose | Tracked in git? |
|----------|---------|-----------------|
| `C:/writers-voice/` (this repo) | Public source. What users install. Pushed to `github.com/travsteward/writers-voice`. | Yes — everything except `voice/` user data |
| `~/.claude/skills/writers-voice/` | Travis's local installed copy. Has personal `voice/` data (corpus, anchor blend, NEVER rules). | No — local only |

When making structural changes (SKILL.md, catalog/*.md, README.md): edit either location, then mirror to the other before committing. The two should stay in sync at all times.

`voice/` is user data and gitignored except for `voice/README.md`, `voice/corpus/.gitkeep`. Travis's actual voice profile (anchor.md, never-rules.md, fingerprints.md, stats.md, status.md, examples.md, corpus/*) lives only in his installed copy.

## Architecture

Two-layer system:

1. **Programmatic anchor** — the web tool at `openwriter.io/writers-voice`. Deterministic feature extraction + Claude-as-matcher running on a Cloudflare Worker. Source lives in the [openwriter-site](https://github.com/travsteward/openwriter-site) repo (private). Produces the author blend the user pastes into their installed skill.

2. **Best-effort everything else** — this skill. The agent reads `catalog/*.md` (instructions on what to look for) and `voice/corpus/*` (user's writing), then writes the output `.md` files in `voice/`. No Node.js, no dependencies, no language runtime. The agent IS the extractor.

The split exists because the anchor matching is the only step that benefits from deterministic feature extraction (sentence distribution, punctuation rates) before the LLM call. Everything downstream (counting AI tells, detecting fingerprints, applying the authenticity hurdle) is fine as agent best-effort — the hurdle thresholds have margin built in to absorb counting drift.

## File Reference

- **`SKILL.md`** — canonical instruction file. What an agent reads when the skill is invoked. The trigger phrases, routing logic, Setup Flow, Anchor Protocol, Analysis Protocol, Apply Protocol all live here. Edit carefully — this is the user-facing contract.
- **`README.md`** — public README on GitHub. Install paths (Claude Code, Vercel skills CLI, manual), Quick Start, architecture overview.
- **`package.json`** — published metadata (name, version, repo URL). Bump `version` when SKILL.md or catalog/ content changes meaningfully.
- **`catalog/`** — read-only reference data the agent loads at analysis time:
  - `ai-tells.md` — 60+ AI words/transitions/phrases/patterns to count against
  - `fingerprints.md` — 8 binary presentation detectors with decision rules
  - `hurdle.md` — authenticity threshold logic (distinguishes user's signature use from AI contamination)
  - `anchor-prompt.md` — stylometry matcher prompt for in-agent anchor generation (skill-mode equivalent of the web tool)
  - `author-hints.md` — curated list of ~70 training-data authors with prose-feature notes
- **`voice/`** — user data (gitignored except README + corpus/.gitkeep):
  - `voice/README.md` — explains what each file in `voice/` does
  - `voice/anchor.md` — the user's author blend (from web tool paste OR generated in-agent)
  - `voice/stats.md`, `voice/never-rules.md`, `voice/fingerprints.md`, `voice/status.md` — auto-generated by the agent
  - `voice/examples.md` — manually curated by the user
  - `voice/corpus/*.md` — raw writing samples the user accumulates

## API Endpoint Naming

The Cloudflare Worker exposes `/api/writers-voice` for the matcher. The KV namespace binding is `VOICE_MATCH_CACHE` (legacy name — kept for binding stability; renaming would require recreating the KV namespace in Cloudflare and migrating cached results).

## Conventions

- **Commits**: prefix with type (`feat:`, `fix:`, `chore:`, `docs:`, `refactor:`). Use `wip:` for in-progress checkpoints. Co-author Claude when relevant.
- **Version bumps**: when SKILL.md or catalog/ content changes meaningfully, bump `package.json` version. Patch for small fixes, minor for new protocols/features.
- **Mirror discipline**: any structural edit goes to BOTH locations (this repo + installed skill) before committing. The two are mechanically identical; only `voice/` differs.
- **Push gates**: never push without explicit user approval. Pushes to this repo just update GitHub; pushes to openwriter-site trigger a live Cloudflare deploy.

## Naming History

This skill went through four names in one session before settling:

1. `voice-presets` → rejected (too generic, conflicts with the actual `voice-presets` skill)
2. `voice-match` → original, paired with openwriter.io/voice-match
3. `voice-anchor` → renamed for the anchor concept, then user reverted
4. **`writers-voice`** → final. Pairs with `authors-voice` (paid plugin) and kills the audio ambiguity that "voice-match" had on aggregator pages with no context.

The GitHub repo (previously `voice-match`) auto-redirects the old URL. The marketing page redirects both `/voice-anchor` and `/voice-match` → `/writers-voice` (301). Internal API endpoint follows the brand: `/api/writers-voice`.

## Learnings

- **Pure markdown skill philosophy**: a skill is guidance for an agent, not a CLI dependency. Resist the temptation to add Node.js scripts. The agent should be the extractor. Counting drift is acceptable when hurdle thresholds have margin.
- **Replace-all hazards**: when renaming brand slugs across files, watch for hyphenated derived words (e.g. `voice-matched` becoming `writers-voiceed`). Verify after replace_all, fix breakage individually.
- **Internal vs user-facing naming**: API endpoints and lib directories that NO external consumer references can be renamed safely along with the brand. Conservative "leave it" is wrong when you control all consumers.
